### Краткое описание проекта
Проект представляет из себя веб-приложение для классификации грибов (ядовитый/неядовитый) для нужд фармкомпании.

### Архитектура
Представляет из себя монолитное веб-приложение со встроенным простым фронтендом. В нем имеется интеграция с Prometheus, а также feature store, сделанный на postgres
 
### Как построен пайплайн
При каждом пуше запускается пайплайн, загружающий данные с фича стора, обрабатывающий данные, обучающий модель по свежим данным, которые берутся из Postgres. Далее с полученной моделью он собирает веб-приложение и "деплоит" его на сервер.

В ходе сборки также создаются артефакты - pkl файлы и файлы образов. Более того, поскольку известна дата запуска пайплайна, его легко можно повторно воспроизвести локально, изменив SQL-запрос, добавив фильтр по дате.

Деплой происходит по схеме Blue-Green, причем для безопасности было принято решение, что переключение инстансов будет делать MLOps вручную. Схема деплоя проста - в зависимости от ветки деплоится либо в blue, либо в green инстанс.

Хотя пайплайн и содержит формальную проверку метрик, я не стал делать, чтобы он падал, если какие-то метрики не соответствуют ожиданиям. Образ (при условии, что не будет пуша сразу в мастер ветку) в любом случае загрузится на неактивную ноду, где ее можно будет дополнительно вручную протестировать и принять решение о переключении Green/Blue. Тем не менее, можно легко добавить "падение" в пайплайн, если метрики не удовлетвояют качеству

P.S: Обычно в своих проектах я использую portainer, и вызываю его метод API, на который отправляю собранный образ и запускаю пересбор стека. В этот раз у меня был очень ограниченный по ресурсам хостинг, поэтому, для простоты я вручную скопировал туда образ и развернул его локально, без привлечения portainer. Будем считать, что в последней "заглушке" пайплайн вызывает API-метод портейнера или пушит в локальный регистр образ.

### Как запустить у себя
Соберите dockefile в папке app, и разверните на порту 8000. После развертывания будут доступны следующие урлы:
- /health - проверка состояния
- /predict - предсказание
- /metrics - метрики
- / - при обращении к корню вернет html-страницу с фронтом для более удобного взаимодействия с моделью
