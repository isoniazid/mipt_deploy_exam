# ML-манифест разработки системы классификации грибов для фармацевтической промышленности

## 1. Предыстория

Заказчиком является фармацевтическая компания, занимающаяся исследованием грибов для исследования их потенциала в медицинской промышленности.

Полное биохимическое исследование гриба - дорогостоящая и длительная процедура, требующая вложения средств и привлечения квалифицированных специалистов. 

Основная проблема заказчика заключается в необходимости быстро отсеивать потенциально токсичные образцы ДО проведения биохимического анализа, чтобы не тратить человеческий и финансовый ресурс на бесперспективные грибы.

---

## 2. Ценностное предложение

Предложенная ML-система позволяет автоматизировать первичный отсев грибов по признаку "ядовитый/неядовитый", основываясь на их внешних характеристиках (шляпка, толщина ножки и тид.), тем самым сокращая объем работы и финансовые вложения на биохимические исследования.

Система создает следующую ценность:
* Ускоряет анализ новых образцов
* Снижает нагрузку на специалистов-биохимиков
* Сокращает расходы на лабораторный анализ
* Повышает эффективность отбора перспективных грибов для фарм/пищевой промышленности 

Следует отметить, что система используется для ПОДДЕРЖКИ принятия решений, а не для замены полноценного исследования. Однозначно сказать, ядовит гриб или нет, а также получить сертификацию препарата мы сможем только после многочисленных лабораторных/доклинических/клинических исследований. Если буквально сформулировать, на какой вопрос отвечает данная система, то это будет "Насколько вероятно, что в процессе лабораторного анализа мы выясним, что этот гриб ядовит, и дальнейшие этапы исследования нужно будет свернуть?"

---

## 3. Цели

Ключевые задачи проекта:

1. Разработать ML-модель для классификации грибковых культур.
2. Автоматизировать предварительную оценку токсичности образцов.
3. Реализовать API для получения предсказаний.
4. Снизить количество потенциально опасных образцов, передаваемых на дальнейшие исследования.
5. Обеспечить возможность накопления и повторного использования новых данных.
6. Реализовать механизм обратной связи от лабораторных специалистов.

---

## 4. Решение

Система представляет собой сервис определения ядовитых грибов на основе табличных характеристик образца

### Используемые признаки:

* текстура поверхности шляпки
* характер распределения спор
* гибкость ножки гриба
* толщина кольца на ножке
* форма шляпки
* поверхность шляпки
* цвет шляпки
* тип прикрепления пластинок
* форма ножки
* тип завесы
* цвет завесы
* количество колец
* тип популяции
* среда обитания

### Интеграции:

* REST API;
* база данных лабораторных исследований;

### Ограничения:

* Система не предполагает мгновенной обработки пользовательских запросов (могут быть задержки, длительность инференса в широком диапазоне)
* Система не предполагает дообучения "на ходу"

### Уровень зрелости ML-системы

Заявляется уровень зрелости **1**.

Система включает:
- Версионирование кода
- CI/CD пайплайн для выкатки модели на прод
- Фича-стор
- Сервинг модели через API
- Мониторинг качества предсказания модели

Полноценный автоматизированный MLOps pipeline отсутствует, хотя есть его суррогатная реализация: скрипт на питоне, который загружает все данные из фича-стора (БД), и на них обучает модель

## 7. Метрики

Из того, что у нас бинарная классификация, следуют две бизнес-метрики и производные от них:
- Чувствительность (доля истинно положительных результатов (TPR), насколько хорошо система определяет действительно ядовитые грибы)
- Специфичность (доля истинно отрицательных результатов (TNR), насколько часто система определяет хорошие грибы как хорошие, а не заносит их в "ядовитые")
- ROC/AUC
- F1 Score

Поскольку заказчик - фармкомпания, и биохимический анализ после проверки будет проводиться в любом случае, чувствительность здесь менеее важна, чем специфичность. Если мы случайно определим ядовитый гриб как безопасный, то заказчик потеряет несколько тысяч долларов на бесполезный биохимический анализ, а если мы "хороший" гриб запишем как ядовитый, то заказчик может потерять миллиарды долларов, не создва из него потенциально эффективный лекарственный препарат. 

Таким образом, ключевая бизнес-метрика - *специфичность* (Specifity/TNR), способность модели корректно определять потенциально полезные грибы, а не классифицировать их как токсичные

## 10. Инференс

Инференс выполняется в реальном времени через REST API.

Причины выбора:
* Количество запросов в секунду небольшое
* Модель относительно простая, ей нечего долго вычислять
* Количество данных, отправляемых лабораториями, небольшое и может быть быстро обработано
* Ждать час/сутки неудобно

## 11. Обратная связь

Источники обратной связи:

MDD (Metric Driven Development) — **Да**.
